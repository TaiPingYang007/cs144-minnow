## 1. 全景：Router 在协议栈哪、和 NetworkInterface 怎么接

```
        ┌─────────────────────────────────────────┐
check6  │   Router —— 路由逻辑（最长前缀匹配）       │   ← 现在在这
        └───┬─────────────┬─────────────┬──────────┘
        ┌───┴───┐     ┌───┴───┐     ┌───┴───┐
check5  │ NetIf │     │ NetIf │     │ NetIf │       ← 复用的网卡，多份
        │ eth0  │     │ eth1  │     │ eth2  │
        └───┬───┘     └───┬───┘     └───┬───┘
         去伦敦         去圣保罗        去芝加哥
```

**比方**：check5 = **一台机器的一张网卡**（一只手，连一段本地链路）。
check6 = **一个有很多张网卡的盒子**（路由器/小区核心交换机）。

🔑 **Router 只碰 IP 层**：不碰 TCP、不碰 ARP、不碰以太网、**不碰 MAC**。
它只干一件事：某网卡收到一个目的地不是自己的 IP 数据报 → 查表决定「从哪张网卡、
发给哪个下一跳」→ 转出去。封帧 / ARP 查 MAC / 排队那一整套**全甩给 check5 的
`send_datagram`**（白嫖）。

**和 check5 的闭环**：check5 笔记里「网卡故意收下不是给自己 IP 的包，推进
`datagrams_received_`」——当时觉得奇怪，**那个奇怪就是为了今天**：路由器的网卡收下
「不是给自己」的包，交给 Router 转发。两个 check 在此闭环。

**接口对接**（`router.hh`）：
- Router 持有 `vector<shared_ptr<NetworkInterface>> interfaces_`，`interface(N)` 取第 N 张。
- **进**：每张网卡的 `datagrams_received()` 队列，堆着它收到的、等转发的数据报。
- **出**：`interface(num)->send_datagram(dgram, next_hop)`。

## 2. 为什么要「最长前缀匹配」(LPM)

**比方：邮局分拣 + 优先级。** 一个目的 IP 可能**同时落进好几个网段**（网段有大有小、
会嵌套），到底听谁的？

> **规则：谁的前缀越长（/N 越大）、网段越小越具体，就听谁的。** 多条匹配时取最长。
> `/0`（`0.0.0.0/0`）= 最笼统的「全世界」= 兜底**默认路由**，只在没别人匹配时才轮到它。

现实同理：「北京海淀区的件」优先于「中国的件」。

## 3. 🔑🔑 前缀匹配真相：按 **bit** 比，不按「字节」比（最大的坑）

**最容易错的认知**：以为 `/17` = 「前两个字节(16 位)一样就行」。**错!**

### 地基：IP 地址其实是 32 个 bit

`143.195.131.17` 看着是 4 个十进制数，**那是给人读的化妆**。机器眼里是 **32 个 0/1**。
每段(0~255) = 8 个 bit。拆十进制→8bit 用**权重减法**（权重 `128 64 32 16 8 4 2 1`，
能减记 1 减掉、不能记 0）：

```
131 = 128 + 2 + 1 = 10000011
143 = 10001111   195 = 11000011   17 = 00010001
```

### `/N` = 「只比最左 N 个 bit，全同才匹配」

手算 `143.195.131.17` 对 `143.195.0.0/17`（反直觉例子，必看）：

```
            bit1~8   bit9~16  bit17↓
目的地    : 10001111 11000011 1|0000011 00010001
/17 规则  : 10001111 11000011 0|0000000 00000000
            └─全同─┘ └─全同─┘ ✗ 第17位:目的=1,规则=0 → 不匹配!
```

前 16 位(143.195)确实一样，**但 /17 要比 17 位，多出来的第 17 位对不上** → **不匹配**!
（131 = `1`0000011，最高位是 1；而 143.195.0.0 第三字节是 0）。

> **字节够了，那多出来的 1 个 bit 不够。** 所以只能数 bit，不能数字节。

对比 `/18`（`143.195.128.0`，128=`10000000`）：比前 18 位 `10` == `10` → **匹配**。
结论：`143.195.131.17` 只匹配 `/18` 和 `/0`，**不匹配 `/17`**；`/18` 更长 → 胜 → 走 hs_router。

**口诀**：① IP = 32 bit ② `/N` = 只比最左 N 位、全同才匹配 ③ 多条匹配取 N 最大。

## 4. mask（掩码）+ `/0` 移位坑（位运算命根子）

代码里不能一位位抠，用**掩码**一刀切：

> 做 `mask`：**最高 N 位是 1、低 (32−N) 位是 0**。目的 IP 和 route_prefix **都 `& mask`**
> （只留高 N 位），相等 ⟺ 高 N 位全同 ⟺ 匹配。

### 拆这行 C++：`~0u << (32 - prefix_length)`

| 符号 | 含义 | 例 |
|---|---|---|
| `0u` | 32 位无符号 0（`u`=unsigned，位运算一律用无符号免符号位捣乱） | 全 0 |
| `~x` | 按位取反 0↔1 | `~0u` = 全 1（造全 1 的惯用法） |
| `x << k` | 左移 k 位，**右边补 0** | `~0u<<8` = 高 24 位 1 |

```
~0u << 8  = 11111111 11111111 11111111 00000000   (N=24, 255.255.255.0)
~0u << 15 = 11111111 11111111 1·0000000 00000000   (N=17, 高17位1)
```

移位量是 `32 − N`：从全 1 出发，把低 (32−N) 位推成 0，剩高 N 位是 1。

### ☠️ `/0` 的坑：`~0u << 32` 是未定义行为(UB)

N=0 → `~0u << (32-0)` = `~0u << 32`。**对 32 位整数移 32 位 = UB**（标准规定移位量 ≥ 位宽
非法）。硬件可能只取移位量低 5 位 → `<<32` 变 `<<0` = **原值不变(全 1)**，而非你以为的全 0
→ 默认路由失效；且 UBSan 会当场抓崩（handout 明说测试在 sanitizer 下跑）。

**解法（特判）**：
```cpp
uint32_t mask = ( prefix_length == 0 ) ? 0u : ( ~0u << ( 32 - prefix_length ) );
```
N==0 直接给全 0（谁都匹配），别让 `<<32` 发生。

## 5. 数据流：帧 → 数据报 → route() → 发送（破「盲人摸象」）

```
        网线上跑【以太网帧】= 信封(MAC) + 信(IP数据报)
                       │ 帧到达某网卡
   ┌────────────── 路由器 Router ───────────────┐
   │ ① recv_frame(frame)  ← check5 干的!          │
   │    拆信封，把【IP数据报】掏出来              │
   │              ▼ 塞进这张网卡的「收件箱」队列   │
   │   [eth0收件箱] [eth1收件箱] [eth2收件箱] ← datagrams_received()
   │ ② route()  ← check6!没参数，主动去掏收件箱   │
   │    读目的IP，查表，TTL-1                      │
   │ ③ send_datagram(dgram, next_hop) ← check5 干的!│
   │    重新封信封(ARP查MAC)，发回网线            │
   └──────────────────────────────────────────────┘
```

🔑 **三个顿悟点**：
1. **接收「帧」在 check5 的 `recv_frame`，不在 route()**。轮到 route() 时帧早被剥掉，
   队列里躺的是**数据报(IP 层)**，route 这层根本不见帧、不见 MAC。
2. **route() 没有参数**——数据不是当场传进来，是**已经躺在各网卡收件箱队列里**。
   route() 是「主动去翻收发室格子取件」(**pull**)，不是「快递员上门塞给你」(push)。
   测试 `simulate()` 循环调 `route()` 256 次 = 定时去翻。
3. **目的 IP = `dgram.header.dst`**（数据报头的字段，uint32_t）。别和「网卡自己的 IP」
   `ip_address_` 混：前者是「包要去哪」，后者是「这张网卡是谁」。

三方法分工：

| 方法 | 属于 | 输入 | 输出 |
|---|---|---|---|
| `recv_frame(frame)` | check5 网卡 | 以太网**帧** | 数据报 → 收件箱队列 |
| `route()` | **check6 路由器** | **无参**（掏队列） | 调 send_datagram |
| `send_datagram(dgram,nh)` | check5 网卡 | 数据报+下一跳 | 以太网**帧** → 网线 |

## 6. `route()` 6 步流水线（最终实现骨架）

```
外层 for: 遍历每张网卡 iface（interfaces_）
  内层 while: 掏空这张网卡的 datagrams_received() 队列（一个不剩）
    对单个数据报:
      ① 取目的 IP:  dgram.header.dst
      ② TTL 不够先丢:  if ttl <= 1 → pop + continue
      ③ 查最长前缀匹配: 遍历 routing_table_，best 记「目前最长的匹配」
                       best 更新条件: best==nullptr || route.prefix_length > best->prefix_length
      ④ 无匹配 → 丢（不发）
      ⑤ 有匹配: ttl--; compute_checksum();   ← 改了TTL必重算校验和!
      ⑥ 定 next_hop 发送:
           next_hop 有值 → 用它（中转路由器IP）
           next_hop 空   → 用 dgram.header.dst（直连，下一跳=最终目的地）
           interfaces_[best->interface_num]->send_datagram(dgram, next_hop)
      ⑦ pop（无论发还是丢，都要从队列移除）
```

> **为什么 while 不是 if**：一张网卡一轮可能堆**多个**待转发包，while 掏空、if 只处理一个。
> **为什么求 best 要带比较**：LPM = 求最值，光「遇匹配就记」会变成「记最后一个匹配」，必须
> `prefix_length > best->prefix_length` 才更新（见踩坑 3）。

## 7. 🔑 next_hop vs interface_num（反复混，必背）

发送这一行 `interfaces_[best->interface_num]->send_datagram(dgram, next_hop)` 里有**两个不同
的东西**，别揉成一个：

| 字段 | 管什么 | 类比 | 代码位置 |
|---|---|---|---|
| `interface_num` | 从**哪张网卡**出去（哪个口） | 走哪个门 | `interfaces_[...]` 选对象 |
| `next_hop`（有值） | 这一跳**交给谁的 IP**（网卡拿它 ARP 查 MAC） | 出门后把包递给哪个人 | `send_datagram` 第 2 参 |

**next_hop 空 vs 有值**：

| next_hop | 含义 | 下一跳填谁 | 网卡 ARP 查谁的 MAC |
|---|---|---|---|
| 空 `{}` | 目的网段**直连**在路由器这张网卡上 | 最终目的地 `dgram.header.dst` | 最终目的地 |
| 有值 | 目的网段不直连，要先交给**下一个路由器** | 那个中转路由器的 IP | 中转路由器 |

## 8. checksum（校验和）答疑

**比方**：包裹封条写「内容校验码」。中转站动了内容（`ttl--`）→ 必须撕旧封条、按新内容重贴，
否则收件人核对不符当损坏拒收。`compute_checksum()` = 用当前 header 重算、覆盖旧值。

- **内部实现黑盒**：IPv4 是「16 位反码求和」，知道「算出校验码填进去」即可。
- **覆盖范围**：**只 IP header，不含 payload**（payload 完整性由上层 TCP/UDP 自己的 checksum 管）。
- **何时重算**：**改了 header 任何字段就要**。路由器转发**只改 TTL**，所以只在 `ttl--` 后算一次；
  `src/dst` IP 不变（呼应 check5「IP 头源/目的 IP 全程不变」）、payload 不动。
- **pop 需不需要**：不需要。pop 是路由器**内部**清队列的动作，不碰数据报字节；且 pop 在
  send 之后，包早发出去了。checksum 只跟「数据报内容变没变」有关。

## 9. 踩坑记录（5 个，下次直接避开）

1. **`std::optional` 缺 `std::`** — `router.hh` 头文件无 `using namespace std`，写
   `optional<Address>` 编译报 `'optional' does not name a type`。**check5 同款坑**（`.hh` 要写全
   `std::`）。后面 `push_back` 的长报错是连锁，修 `std::` 自动消失。
2. **`struct` 成员加 `const` → 删掉赋值能力** — `const` 成员让 `operator=` 被删除，vector 不能
   sort/erase。**单独不报错**，但配 `map::operator[]`（需默认构造）就触发 `-Werror=effc++`
   （成员未在初始化列表初始化）。去掉 const + 成员加 `{}` 默认初始化（**check5 同款 -Weffc++ 坑**）。
3. **map→best 换写法漏了 prefix_length 比较** — 最初用 `map<prefix_length, RouteEntry>` +
   `rbegin()` 选最长（能用但绕、低效、还触发坑 2）。换成指针 `best` 遍历时，**漏了 `>` 比较**
   写成无条件 `best = &route`，变成「记最后一个匹配」→ 192.168.0.2 被错发默认路由。
   **求最值经典坑：遍历 + 比较缺一不可。**
4. **漏 `compute_checksum()`** — `ttl--` 后没重算校验和。测试 `equal()` 比的是 **serialize 后的
   字节**（含 header checksum），旧 checksum 对应 ttl=64、期望对应 ttl=63 → 字节不等 → 失败。
5. **过程教训**：一度直接问 AI 写 `route()`，AI 代码**连编译都没过**（坑 2）**且漏 checksum**（坑 4）
   ——核心逻辑自己写/改才不埋雷、才真懂。

## 自测题（第 1 轮 · 全景 + LPM）

1. Router 在协议栈哪一层？它碰 MAC / ARP / 以太网帧吗？封帧那套谁干的？
2. 为什么要最长前缀匹配？`/0` 是什么角色？
3. `143.195.131.17` 匹配 `143.195.0.0/17` 吗？逐 bit 说明为什么。
4. `/24` 的掩码二进制长啥样？`~0u << (32-24)` 怎么算出来的？

<details><summary>参考要点</summary>

1. IP（网络）层，在 NetworkInterface 之上。**不碰** MAC/ARP/以太网帧——封帧、ARP 查 MAC 全甩给
   check5 的 `send_datagram`。Router 只读 IP 数据报、查表、改 TTL。
2. 一个目的 IP 可能落进多个嵌套网段，取最具体（前缀最长）那条。`/0` = 兜底默认路由，没别人匹配才用。
3. **不匹配**。前 16 位(143.195)同，但 /17 要比第 17 位：目的地 `1`（131=`1`0000011）≠ 规则 `0`
   （第三字节 0）→ 前 17 位没全同 → 不匹配。只匹配 /18 和 /0。
4. `11111111 11111111 11111111 00000000`(255.255.255.0)。`~0u`=全1，左移 8 位、右补 8 个 0 → 高 24 位 1。

</details>

## 自测题（第 2 轮 · 数据流 + route 流程）

1. `route()` 为什么没有参数？它处理的数据报从哪来？帧在哪个方法里被拆掉？
2. 内层为什么用 `while` 不用 `if`？外层 `for` 又遍历什么？
3. `best_route` 怎么选出来？只「遍历」够不够？缺什么会发生什么 bug？
4. `next_hop` 空和有值，分别在 `send_datagram` 第二个参数填谁？和 `interface_num` 什么区别？

<details><summary>参考要点</summary>

1. 数据不是当场传进来，是已躺在各网卡 `datagrams_received()` 队列里（pull 模式，主动掏）。帧在
   check5 的 `recv_frame` 就拆成数据报入队了，route() 只见数据报。
2. 一张网卡一轮可能堆多个待转发包，`while` 掏空；`if` 只处理一个。`for` 遍历所有网卡。
3. 遍历 routing_table_ + **比较 prefix_length** 记最长匹配。光遍历不够——漏了 `>` 比较会变成
   「记最后一个匹配」，导致选到错误（更短）的路由（如把 192.168.0.2 发去默认路由）。
4. 空 → 填 `dgram.header.dst`（直连，下一跳=最终目的地）；有值 → 填那个中转路由器 IP。
   `interface_num` 管「从哪张网卡出」，`next_hop` 管「这一跳交给谁的 IP」，两回事。

</details>

## 自测题（第 3 轮 · 踩坑 + 概念）

1. `ttl--` 之后为什么必须 `compute_checksum()`？checksum 覆盖到 payload 吗？`pop` 要不要重算？
   ttl数据数据报内的内容，为了确保传输路径和次数准确，所以需要重新校验，供下一跳检查
   checksum 只能覆盖到 header
   pop不要重新算，pop是整个删除了，不是修改数据报内容

2. `/0` 为什么不能直接 `~0u << 32`？会发生什么？怎么绕？
   ~0u 表示 32位 全是1
   ~0u << 32 是未定义行为(UB)——C++ 标准规定:对 32 位整数移位 ≥ 32 位是非法的。
   如果是0 就设置 0u

3. 为什么 `struct RouteEntry` 成员加 `const` + 用 `map[k]=v` 会编译失败？
   因为`map[k]=v`如果不存在就会先创建新值再存入，创建包含了初始化
    →  -Weffc++ 检查这个默认构造: 要求所有成员在初始化列表里初始化
    →  RouteEntry 成员既没 {} 也没初始化列表  →  报错!
     (error: 'route_prefix' should be initialized in the member initialization list)

<details><summary>参考要点</summary>

1. 改了 header（TTL）旧 checksum 就失效，下一跳核对不符会丢包，必须重算。**只覆盖 header、不含
   payload**。pop 是内部清队列、不碰字节，不需要重算。
2. `<<32` 对 32 位整数是 UB；硬件可能只取低 5 位 → `<<0` → 原值不变(全1)，默认路由失效；UBSan 抓崩。
   特判 `prefix_length==0 ? 0u : ...`。
3. `const` 成员删掉 `operator=`；`map::operator[]` 需要先默认构造一个 `RouteEntry()`，而合成默认
   构造没在初始化列表初始化成员 → `-Werror=effc++` 报错。去 const + 成员 `{}`，或干脆别用 map。

</details>
